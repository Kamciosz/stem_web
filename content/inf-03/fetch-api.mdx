`fetch()` pozwala przeglądarce rozmawiać z serwerem — pobierać dane, wysyłać formularze, ładować kolejne części strony bez przeładowania. W tej lekcji nauczysz się podstaw asynchroniczności, `fetch`, `async/await` i obsługi błędów.

## Czym jest asynchroniczność

JavaScript domyślnie wykonuje kod linia po linii, synchronicznie. Ale niektóre operacje (pobieranie danych, timery) trwają długo — gdyby czekać, strona by "zamroziła". Asynchroniczność pozwala uruchomić taką operację i kontynuować pracę, a wynik odebrać gdy będzie gotowy.

```js
// Synchronicznie — zatrzymuje na 2 sekundy
const data = pobierzDane();        // czeka 2s
console.log("Dane:", data);        // dopiero teraz
console.log("Koniec");             // 2s opóźnienia

// Asynchronicznie — NIE czeka
pobierzDane().then(data => {
    console.log("Dane:", data);
});
console.log("Koniec");             // wyświetli się NATYCHMIAST
```

## Promise — obietnica wyniku

`fetch()` zwraca **Promise** — obiekt, który reprezentuje przyszły wynik. Promise ma 3 stany:

- **pending** — czeka na wynik.
- **fulfilled** — sukces (mamy dane).
- **rejected** — błąd (np. serwer nie odpowiada).

```js
fetch("https://api.example.com/users")    // → Promise
    .then(response => response.json())     // .then() obsługuje sukces
    .then(data => console.log(data))       // kolejny .then() przetwarza dane
    .catch(error => console.error(error)); // .catch() obsługuje błąd
```

## Pierwszy fetch

```js
fetch("https://jsonplaceholder.typicode.com/users/1")
    .then(response => {
        if (!response.ok) {
            throw new Error(`HTTP ${response.status}`);
        }
        return response.json();
    })
    .then(user => {
        console.log(user.name);
        console.log(user.email);
    })
    .catch(error => {
        console.error("Nie udało się pobrać:", error);
    });
```

<Callout type="warning" title="fetch NIE rzuca na 404/500">
`fetch()` rzuca odrzucenie (reject) tylko przy błędach sieci. HTTP 404 czy 500 to "udana" odpowiedź — musisz sam sprawdzić `response.ok` (true gdy status 200–299).
</Callout>

## `async / await` — czytelniejsza składnia

Zamiast łańcucha `.then()` możesz pisać kod, który **wygląda synchronicznie**:

```js
async function pobierzUsera(id) {
    try {
        const response = await fetch(`https://api.example.com/users/${id}`);

        if (!response.ok) {
            throw new Error(`HTTP ${response.status}`);
        }

        const user = await response.json();
        return user;
    } catch (error) {
        console.error("Błąd:", error);
        return null;
    }
}

const user = await pobierzUsera(1);
console.log(user?.name);
```

`async` oznacza funkcję, która zwraca Promise. `await` czeka na Promise i zwraca wartość (lub rzuca błąd).

### Top-level await

W modułach ES (`<script type="module">`) możesz używać `await` na najwyższym poziomie:

```js
// app.js (moduł)
const response = await fetch("/api/data");
const data = await response.json();
console.log(data);
```

W klasycznym skrypcie (nie module) musisz owinąć w `async` funkcję.

## Wysyłanie danych — POST, PUT, DELETE

```js
async function utworzUsera(dane) {
    try {
        const response = await fetch("/api/users", {
            method: "POST",
            headers: {
                "content-type": "application/json"
            },
            body: JSON.stringify(dane)
        });

        if (!response.ok) {
            throw new Error(`HTTP ${response.status}`);
        }

        return await response.json();
    } catch (error) {
        console.error("Błąd:", error);
        return null;
    }
}

utworzUsera({ imie: "Anna", email: "anna@example.com" });
```

Właściwości `fetch(url, options)`:

- **`method`** — `GET` (domyślne), `POST`, `PUT`, `PATCH`, `DELETE`.
- **`headers`** — dodatkowe nagłówki (np. `content-type`, `authorization`).
- **`body`** — dane do wysłania (string, FormData, Blob, JSON).
- **`credentials`** — `include` jeśli chcesz wysyłać ciasteczka cross-origin.
- **`signal`** — AbortSignal do anulowania requestu.

## Przykład — wyświetlanie listy z API

```html
<ul id="users"></ul>
<button id="reload">Odśwież</button>
```

```js
const list = document.querySelector("#users");
const btn = document.querySelector("#reload");

async function renderUsers() {
    list.innerHTML = "<li>Ładowanie…</li>";

    try {
        const response = await fetch("https://jsonplaceholder.typicode.com/users");
        if (!response.ok) throw new Error(`HTTP ${response.status}`);

        const users = await response.json();

        list.innerHTML = "";   // wyczyść "Ładowanie…"
        for (const user of users) {
            const li = document.createElement("li");
            li.textContent = `${user.name} (${user.email})`;
            list.appendChild(li);
        }
    } catch (error) {
        list.innerHTML = `<li class="error">Błąd: ${error.message}</li>`;
    }
}

btn.addEventListener("click", renderUsers);
renderUsers();   // od razu po załadowaniu
```

## Promise.all — równoległe zapytania

Gdy potrzebujesz danych z kilku endpointów naraz:

```js
async function pobierzWszystko() {
    try {
        const [users, posts, comments] = await Promise.all([
            fetch("/api/users").then(r => r.json()),
            fetch("/api/posts").then(r => r.json()),
            fetch("/api/comments").then(r => r.json())
        ]);

        console.log(`${users.length} użytkowników, ${posts.length} postów`);
    } catch (error) {
        console.error("Jeden z endpointów zawiódł:", error);
    }
}
```

`Promise.all` czeka na wszystkie. **Jeden błąd = cały Promise rzuca**. Użyj `Promise.allSettled` jeśli chcesz wynik nawet przy częściowym błędzie.

## AbortController — anulowanie requestu

Gdy user anuluje wyszukiwanie (klawisz Escape, nowe litery w polu), warto przerwać stary request:

```js
let controller = null;

async function szukaj(query) {
    // Anuluj poprzedni
    if (controller) controller.abort();
    controller = new AbortController();

    try {
        const response = await fetch(`/api/search?q=${query}`, {
            signal: controller.signal
        });
        return await response.json();
    } catch (error) {
        if (error.name === "AbortError") {
            console.log("Wyszukiwanie anulowane");
        } else {
            throw error;
        }
    }
}
```

## CORS — problem cross-origin

Przeglądarka domyślnie blokuje fetch do innej domeny niż strona (np. strona na `example.com` nie odpytuje API na `api.example.org`). To **CORS** (Cross-Origin Resource Sharing).

```js
// ❌ Zablokowane — inna domena bez pozwolenia
fetch("https://api.inna-domena.pl/data");

// ✅ OK — ten sam origin
fetch("/api/data");

// ✅ OK — API ma nagłówek Access-Control-Allow-Origin
// (konfigurowane po stronie serwera)
```

Jeśli musisz testować bez serwera CORS, włącz w przeglądarce (Chrome: `--disable-web-security`, ale tylko do developmentu).

## Podsumowanie

`fetch(url, options)` zwraca Promise. Pierwszy `.then` ma `Response` z `.json()` / `.text()` / `.blob()`. `async/await` zamienia łańcuch `.then()` w czytelny kod synchroniczny. **ZAWSZE** sprawdzaj `response.ok` — fetch NIE rzuca na 4xx/5xx. `POST`/`PUT`/`DELETE` wymagają `method` + `headers` + `body` (często `JSON.stringify`). `Promise.all` czeka na wiele requestów równolegle. `AbortController` anuluje zbędne requesty. CORS to ograniczenie cross-origin — po stronie klienta nic nie zrobisz, musi odpowiedzieć serwer.
