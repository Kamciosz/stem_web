Wyobraź sobie, że otwierasz aplikację pogodową. Wpisujesz miasto i po chwili widzisz temperaturę, wiatr, prognozę na tydzień. Skąd te dane? Nie były zakodowane w HTML — aplikacja właśnie zapytała serwer i dostała odpowiedź. Bez przeładowania strony, bez czekania. To właśnie robi <Term word="fetch">fetch</Term>.

`fetch()` to wbudowana funkcja JavaScript, która pozwala przeglądarce rozmawiać z <Term word="serwer">serwerem</Term> — pobierać dane, wysyłać formularze, ładować kolejne fragmenty strony. Żeby to zrozumieć, musisz najpierw pojąć jedną rzecz: asynchroniczność.

## Dlaczego JavaScript nie może po prostu czekać

JavaScript w przeglądarce działa w jednym wątku. Gdy wykonuje kod, robi to linia po linii. Gdyby przy każdym zapytaniu do serwera zatrzymał się i czekał na odpowiedź — strona by zamarzła. Nie dałoby się kliknąć, przewinąć, nic.

Dlatego operacje, które trwają — pobieranie danych, timery, odczyt pliku — są asynchroniczne. Uruchamiasz je i mówisz: „zrób to, a gdy skończysz, wywołaj tę funkcję". Tymczasem reszta kodu działa dalej.

```js
// Synchronicznie — zatrzymuje na 2 sekundy
const data = pobierzDane();        // czeka 2s
console.log("Dane:", data);        // dopiero teraz
console.log("Koniec");             // 2s opóźnienia

// Asynchronicznie — NIE czeka
pobierzDane().then(data => {
    console.log("Dane:", data);
});
console.log("Koniec");             // wyświetli się NATYCHMIAST
```

W wersji asynchronicznej `"Koniec"` pojawi się przed danymi — bo przeglądarka nie czeka na odpowiedź serwera, tylko jedzie dalej.

## Promise — obietnica, że coś się stanie

`fetch()` zwraca <Term word="Promise">Promise</Term> — obiekt reprezentujący wynik operacji, która jeszcze się nie zakończyła. Możesz myśleć o nim jak o paragonie ze sklepu: dostajesz go od razu, ale towar odbierzesz za chwilę.

Promise ma trzy stany:

- **pending** — czeka na wynik (towar w drodze).
- **fulfilled** — sukces, mamy dane (towar odebrany).
- **rejected** — błąd, coś poszło nie tak (towar zaginął).

```js
fetch("https://api.example.com/users")    // → Promise
    .then(response => response.json())     // .then() obsługuje sukces
    .then(data => console.log(data))       // kolejny .then() przetwarza dane
    .catch(error => console.error(error)); // .catch() obsługuje błąd
```

Łańcuch `.then()` czyta się jak przepis: „pobierz dane, potem zamień na JSON, potem wyświetl. Jeśli cokolwiek się posypie — złap błąd".

## Pierwszy fetch — krok po kroku

Zacznijmy od prawdziwego przykładu z publicznym <Term word="API">API</Term>:

```js
fetch("https://jsonplaceholder.typicode.com/users/1")
    .then(response => {
        if (!response.ok) {
            throw new Error(`HTTP ${response.status}`);
        }
        return response.json();
    })
    .then(user => {
        console.log(user.name);
        console.log(user.email);
    })
    .catch(error => {
        console.error("Nie udało się pobrać:", error);
    });
```

Zwróć uwagę na `response.ok` — to ważne.

<Callout type="warning" title="fetch NIE rzuca błędu na 404 ani 500">
`fetch()` odrzuca Promise (reject) tylko przy błędach sieci — gdy serwer w ogóle nie odpowiada. Odpowiedź HTTP 404 (nie znaleziono) czy 500 (błąd serwera) to dla `fetch` „udana" odpowiedź. Musisz sam sprawdzić `response.ok` — jest `true` tylko gdy status to 200–299. Bez tego sprawdzenia Twój kod będzie próbował przetworzyć stronę błędu jako dane.
</Callout>

## `async / await` — kod, który wygląda normalnie

Łańcuchy `.then()` działają, ale przy złożonej logice robią się trudne do czytania. Dlatego masz <Term word="async/await">async/await</Term> — składnię, która sprawia, że asynchroniczny kod wygląda jak synchroniczny.

```js
async function pobierzUsera(id) {
    try {
        const response = await fetch(`https://api.example.com/users/${id}`);

        if (!response.ok) {
            throw new Error(`HTTP ${response.status}`);
        }

        const user = await response.json();
        return user;
    } catch (error) {
        console.error("Błąd:", error);
        return null;
    }
}

const user = await pobierzUsera(1);
console.log(user?.name);
```

`async` przed funkcją mówi: „ta funkcja zwraca Promise". `await` przed operacją mówi: „poczekaj na wynik, zanim przejdziesz dalej". Błędy łapiesz przez `try / catch` — tak samo jak przy synchronicznym kodzie.

Efekt: kod czyta się z góry na dół, linia po linii, jak zwykły skrypt. Żadnych zagnieżdżonych callbacków.

### Top-level await — w modułach

W modułach ES (`<script type="module">`) możesz używać `await` na najwyższym poziomie, bez owijania w funkcję:

```js
// app.js (moduł)
const response = await fetch("/api/data");
const data = await response.json();
console.log(data);
```

W klasycznym skrypcie (nie module) musisz owinąć w `async` funkcję — `await` poza nią to błąd składni.

## Wysyłanie danych — POST, PUT, DELETE

`fetch` domyślnie wysyła żądanie <Term word="GET">GET</Term> — pobiera dane. Gdy chcesz coś wysłać, musisz podać opcje:

```js
async function utworzUsera(dane) {
    try {
        const response = await fetch("/api/users", {
            method: "POST",
            headers: {
                "content-type": "application/json"
            },
            body: JSON.stringify(dane)
        });

        if (!response.ok) {
            throw new Error(`HTTP ${response.status}`);
        }

        return await response.json();
    } catch (error) {
        console.error("Błąd:", error);
        return null;
    }
}

utworzUsera({ imie: "Anna", email: "anna@example.com" });
```

Właściwości drugiego argumentu `fetch(url, options)`:

- **`method`** — `GET` (domyślne), `POST`, `PUT`, `PATCH`, `DELETE`.
- **`headers`** — dodatkowe nagłówki (np. `content-type`, `authorization`).
- **`body`** — dane do wysłania. Obiekty JS musisz zamienić na <Term word="JSON">JSON</Term> przez `JSON.stringify()`.
- **`credentials`** — `include` jeśli chcesz wysyłać ciasteczka cross-origin.
- **`signal`** — AbortSignal do anulowania żądania.

<Callout type="info" title="JSON.stringify i JSON.parse">
`JSON.stringify(obiekt)` zamienia obiekt JS na tekst JSON — żeby go wysłać. `JSON.parse(tekst)` zamienia tekst JSON z powrotem na obiekt JS — żeby go użyć. `response.json()` robi to drugie automatycznie.
</Callout>

## Przykład — wyświetlanie listy z API

Połączmy wszystko: fetch, async/await, obsługę błędów i manipulację <Term word="DOM">DOM</Term>:

```html
<ul id="users"></ul>
<button id="reload">Odśwież</button>
```

```js
const list = document.querySelector("#users");
const btn = document.querySelector("#reload");

async function renderUsers() {
    list.innerHTML = "<li>Ładowanie…</li>";

    try {
        const response = await fetch("https://jsonplaceholder.typicode.com/users");
        if (!response.ok) throw new Error(`HTTP ${response.status}`);

        const users = await response.json();

        list.innerHTML = "";   // wyczyść "Ładowanie…"
        for (const user of users) {
            const li = document.createElement("li");
            li.textContent = `${user.name} (${user.email})`;
            list.appendChild(li);
        }
    } catch (error) {
        list.innerHTML = `<li class="error">Błąd: ${error.message}</li>`;
    }
}

btn.addEventListener("click", renderUsers);
renderUsers();   // od razu po załadowaniu
```

Zwróć uwagę na trzy rzeczy: pokazujesz „Ładowanie…" zanim dane przyjdą, obsługujesz błąd wyświetlając komunikat zamiast pustej listy, i używasz `textContent` (nie `innerHTML`) do wstawiania danych od serwera — bezpieczniej.

## Promise.all — kilka zapytań naraz

Gdy potrzebujesz danych z kilku <Term word="endpoint">endpointów</Term> jednocześnie, nie czekaj na nie kolejno — odpytaj je równolegle:

```js
async function pobierzWszystko() {
    try {
        const [users, posts, comments] = await Promise.all([
            fetch("/api/users").then(r => r.json()),
            fetch("/api/posts").then(r => r.json()),
            fetch("/api/comments").then(r => r.json())
        ]);

        console.log(`${users.length} użytkowników, ${posts.length} postów`);
    } catch (error) {
        console.error("Jeden z endpointów zawiódł:", error);
    }
}
```

`Promise.all` czeka na wszystkie Promise i zwraca tablicę wyników. Ważne: **jeden błąd = cały Promise rzuca**. Jeśli chcesz wynik nawet przy częściowym błędzie, użyj `Promise.allSettled` — zwraca status każdego Promise osobno.

## AbortController — anulowanie zbędnych zapytań

Użytkownik wpisuje w wyszukiwarce „java" — wysyłasz zapytanie. Zanim odpowiedź wróci, dopisuje „script" — wysyłasz kolejne. Stare zapytanie jest już niepotrzebne. Możesz je anulować:

```js
let controller = null;

async function szukaj(query) {
    // Anuluj poprzednie zapytanie
    if (controller) controller.abort();
    controller = new AbortController();

    try {
        const response = await fetch(`/api/search?q=${query}`, {
            signal: controller.signal
        });
        return await response.json();
    } catch (error) {
        if (error.name === "AbortError") {
            console.log("Wyszukiwanie anulowane");
        } else {
            throw error;
        }
    }
}
```

`AbortController` tworzy sygnał, który przekazujesz do `fetch`. Gdy wywołasz `.abort()`, fetch natychmiast rzuca `AbortError`. Sprawdzasz nazwę błędu, żeby odróżnić anulowanie od prawdziwego problemu.

## CORS — gdy serwer mówi „nie"

Przeglądarka domyślnie blokuje `fetch` do innej domeny niż ta, na której działa strona. To mechanizm bezpieczeństwa zwany CORS (Cross-Origin Resource Sharing).

```js
// Zablokowane — inna domena bez pozwolenia
fetch("https://api.inna-domena.pl/data");

// OK — ten sam origin
fetch("/api/data");

// OK — API ma nagłówek Access-Control-Allow-Origin
// (konfigurowane po stronie serwera)
```

Jeśli widzisz błąd CORS w konsoli, problem jest po stronie serwera — musi dodać odpowiedni nagłówek. Po stronie <Term word="client-side">klienta</Term> nic nie zrobisz. Podczas developmentu możesz uruchomić własny serwer proxy albo użyć publicznego API, które CORS ma skonfigurowany.

<Callout type="exam">
Na egzaminie INF.03 z fetch sprawdzają: poprawne użycie `async/await` z `try/catch`, sprawdzenie `response.ok` przed `.json()`, właściwe opcje dla POST (`method`, `headers`, `body` z `JSON.stringify`). Pamiętaj, że `fetch` nie rzuca błędu na 404 — to klasyczna pułapka egzaminacyjna.
</Callout>

## Podsumowanie

`fetch(url, options)` zwraca <Term word="Promise">Promise</Term>. Pierwsza odpowiedź to obiekt `Response` — wywołujesz na nim `.json()`, `.text()` albo `.blob()`, żeby dostać dane. <Term word="async/await">async/await</Term> zamienia łańcuch `.then()` w czytelny kod synchroniczny. Zawsze sprawdzaj `response.ok` — `fetch` nie rzuca na 4xx/5xx. <Term word="POST">POST</Term>/PUT/DELETE wymagają `method` + `headers` + `body` (często `JSON.stringify`). `Promise.all` czeka na wiele zapytań równolegle. `AbortController` anuluje zbędne zapytania. CORS to ograniczenie cross-origin — po stronie klienta nic nie zrobisz, musi odpowiedzieć serwer.
