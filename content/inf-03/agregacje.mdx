Wyobraź sobie, że jesteś właścicielem sklepu i chcesz wiedzieć: ile zamówień złożył każdy klient? Kto wydał najwięcej? Jaki był przychód w zeszłym miesiącu? Nie potrzebujesz każdego wiersza z osobna — potrzebujesz **podsumowania**. Do tego służą agregacje: funkcje, które "zwijają" wiele wierszy w jedną wartość.

## Funkcje agregujące — pięć podstawowych

<Term word="SQL">SQL</Term> ma pięć klasycznych funkcji agregujących:

```sql
SELECT
    COUNT(*)              AS liczba_rekordow,  -- liczba wierszy
    COUNT(email)          AS z_emailem,        -- z niepustym emailem
    COUNT(DISTINCT email) AS unikalne_emaile,  -- unikalne wartości
    SUM(amount)           AS suma,             -- suma wartości
    AVG(amount)           AS srednia,          -- średnia
    MIN(amount)           AS minimum,          -- minimum
    MAX(amount)           AS maksimum          -- maksimum
FROM orders;
```

Przykład z danymi:

```
orders
+----+---------+--------+
| id | user_id | amount |
+----+---------+--------+
|  1 |    1    | 50.00  |
|  2 |    1    | 30.00  |
|  3 |    2    | 120.00 |
|  4 |    2    | 80.00  |
|  5 |    3    | 200.00 |
+----+---------+--------+
```

```sql
SELECT
    COUNT(*)      AS liczba_zamowien,   -- 5
    COUNT(amount) AS z_kwota,           -- 5 (amount nie ma NULL)
    SUM(amount)   AS suma,              -- 480.00
    AVG(amount)   AS srednia,           -- 96.00
    MIN(amount)   AS minimum,           -- 30.00
    MAX(amount)   AS maksimum           -- 200.00
FROM orders;
```

<Callout type="tip" title="COUNT(*) vs COUNT(kolumna)">
- `COUNT(*)` liczy wszystkie wiersze, nawet z NULL-ami w kolumnach.
- `COUNT(kolumna)` liczy wiersze, gdzie ta kolumna NIE jest NULL.
- `COUNT(DISTINCT kolumna)` liczy unikalne wartości (bez NULL-i).
</Callout>

## GROUP BY — grupowanie wyników

Samo `COUNT(*)` daje jedną liczbę dla całej tabeli. `GROUP BY` dzieli wiersze na grupy i stosuje agregację do każdej grupy osobno. Wynik: jeden wiersz na grupę.

```sql
SELECT
    user_id,
    COUNT(*)    AS liczba_zamowien,
    SUM(amount) AS suma_zamowien
FROM orders
GROUP BY user_id
ORDER BY suma_zamowien DESC;
```

Wynik:

| user_id | liczba_zamowien | suma_zamowien |
|---------|-----------------|---------------|
| 3       | 1               | 200.00        |
| 2       | 2               | 200.00        |
| 1       | 2               | 80.00         |

### GROUP BY z JOIN — praktyczny przykład

Chcesz zobaczyć imiona klientów zamiast samych ID? Połącz z tabelą klientów przez <Term word="JOIN">JOIN</Term>.

```sql
SELECT
    customers.name,
    COUNT(orders.id)   AS liczba_zamowien,
    SUM(orders.amount) AS suma,
    AVG(orders.amount) AS srednia
FROM customers
LEFT JOIN orders ON customers.id = orders.customer_id
GROUP BY customers.id, customers.name
ORDER BY suma DESC;
```

Wynik:

| name  | liczba_zamowien | suma   | srednia |
|-------|-----------------|--------|---------|
| Jan   | 2               | 200.00 | 100.00  |
| Ola   | 1               | 200.00 | 200.00  |
| Anna  | 2               | 80.00  | 40.00   |
| Piotr | 0               | NULL   | NULL    |  ← LEFT JOIN pokazuje też klientów bez zamówień

## HAVING — filtrowanie po grupowaniu

`WHERE` filtruje wiersze **przed** grupowaniem. `HAVING` filtruje **grupy po** agregacji. To kluczowa różnica.

```sql
-- Tylko klienci z więcej niż 1 zamówieniem
SELECT
    user_id,
    COUNT(*) AS liczba,
    SUM(amount) AS suma
FROM orders
GROUP BY user_id
HAVING COUNT(*) > 1
ORDER BY suma DESC;
```

```sql
-- Tylko kategorie, w których wartość magazynu > 1000
SELECT
    category,
    SUM(price * stock) AS wartosc_magazynu
FROM products
GROUP BY category
HAVING wartosc_magazynu > 1000;
```

<Callout type="warning" title="WHERE vs HAVING">
- **WHERE** — przed grupowaniem, na surowych wierszach. Nie może używać funkcji agregujących.
- **HAVING** — po grupowaniu, na zagregowanych grupach. Może używać `COUNT`, `SUM` itd.

```sql
-- ❌ Błąd: WHERE nie może używać agregacji
SELECT user_id, SUM(amount) FROM orders WHERE SUM(amount) > 100 GROUP BY user_id;

-- ✅ OK: HAVING filtruje po grupowaniu
SELECT user_id, SUM(amount) FROM orders GROUP BY user_id HAVING SUM(amount) > 100;
```
</Callout>

## Kolejność klauzul SQL — zapamiętaj raz na zawsze

SQL wymaga ścisłej kolejności. Nie możesz jej zmienić.

```
SELECT kolumny, agregacje
FROM tabela
[JOIN ... ON ...]
WHERE filtr_wierszy          ← przed grupowaniem
GROUP BY kolumny
HAVING filtr_grup            ← po grupowaniu
ORDER BY kolumny [ASC|DESC]
LIMIT N OFFSET M;
```

## Przykład — raport sprzedaży miesięcznie

```sql
SELECT
    DATE_FORMAT(created_at, '%Y-%m') AS miesiac,
    COUNT(*)                          AS liczba_zamowien,
    SUM(amount)                       AS przychod,
    AVG(amount)                       AS srednia_zamowienie,
    COUNT(DISTINCT customer_id)       AS unikalni_klienci
FROM orders
WHERE status = 'paid'
  AND created_at >= '2026-01-01'
GROUP BY DATE_FORMAT(created_at, '%Y-%m')
HAVING przychod > 0
ORDER BY miesiac DESC;
```

Funkcje daty (`DATE_FORMAT`, `YEAR()`, `MONTH()`) są specyficzne dla MySQL. W PostgreSQL byłoby `TO_CHAR(created_at, 'YYYY-MM')`.

## Podzapytania (subqueries)

Czasem potrzebujesz wyniku jednego zapytania jako wejścia do drugiego. To właśnie podzapytanie — <Term word="query">zapytanie</Term> w zapytaniu.

### Skalarne — zwraca jedną wartość

```sql
SELECT name
FROM customers
WHERE id = (SELECT customer_id FROM orders ORDER BY amount DESC LIMIT 1);
```

Wewnętrzne zapytanie: ID klienta z największym zamówieniem. Zewnętrzne: imię tego klienta.

### W FROM — derived table

```sql
SELECT AVG(suma_per_user) AS srednia_suma_wszystkich_userow
FROM (
    SELECT user_id, SUM(amount) AS suma_per_user
    FROM orders
    GROUP BY user_id
) AS per_user;
```

### W IN — czy wartość jest w zbiorze

```sql
SELECT name FROM customers
WHERE id IN (SELECT DISTINCT customer_id FROM orders WHERE amount > 100);
```

Klienci, którzy złożyli zamówienie powyżej 100.

### EXISTS — czy podzapytanie zwraca cokolwiek

```sql
SELECT name FROM customers c
WHERE EXISTS (
    SELECT 1 FROM orders o WHERE o.customer_id = c.id AND o.amount > 100
);
```

Dla każdego klienta sprawdza: czy istnieje choć jedno jego zamówienie > 100? `EXISTS` jest często szybsze niż `IN` przy dużych zbiorach.

## WITH (CTE) — czytelniejsze podzapytania

CTE (Common Table Expression) to nazwane podzapytanie. Zamiast zagnieżdżać zapytania w `FROM`, definiujesz je na górze i używasz jak tymczasowej tabeli. Dostępne od MySQL 8+.

```sql
WITH top_customers AS (
    SELECT
        customer_id,
        SUM(amount) AS total
    FROM orders
    WHERE status = 'paid'
    GROUP BY customer_id
    HAVING total > 500
)
SELECT
    c.name,
    tc.total
FROM top_customers tc
JOIN customers c ON tc.customer_id = c.id
ORDER BY tc.total DESC;
```

`WITH top_customers AS (...)` definiuje "tymczasową tabelę" do użycia w głównym zapytaniu. Czytelniejsze niż podzapytania zagnieżdżone w `FROM`.

## UNION — łączenie wyników zapytań

```sql
SELECT name, email FROM customers
UNION
SELECT name, email FROM newsletter_subscribers;
```

`UNION` łączy wyniki dwóch `SELECT` (kolumny muszą mieć zgodne typy i tę samą liczbę). `UNION ALL` zachowuje duplikaty i jest szybsze.

<Callout type="exam">
Na egzaminie INF.03 najczęściej pytają o: różnicę `WHERE` vs `HAVING`, poprawną kolejność klauzul (`GROUP BY` przed `HAVING`), `COUNT(*)` vs `COUNT(kolumna)`, oraz `COUNT(DISTINCT ...)`. Naucz się tych czterech rzeczy na pamięć.
</Callout>

## Podsumowanie

`COUNT`, `SUM`, `AVG`, `MIN`, `MAX` to pięć funkcji agregujących. `GROUP BY` dzieli wiersze na grupy — jeden wiersz wyniku na grupę. `WHERE` filtruje przed grupowaniem, `HAVING` po. Kolejność klauzul: `SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT`. Podzapytania w `WHERE` / `FROM` / `SELECT` pozwalają budować złożone zapytania. `EXISTS` często szybsze niż `IN`. `WITH ... AS` (CTE) to czytelniejsza alternatywa dla podzapytań w `FROM`. `UNION` łączy wyniki z dwóch zapytań.
