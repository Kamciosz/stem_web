W prawdziwej bazie danych rzadko pracujesz na jednej tabeli. Userzy, zamówienia, produkty, kategorie — to osobne tabele, połączone **relacjami**. W tej lekcji nauczysz się: kluczy obcych, typów relacji i magicznego słowa `JOIN`.

## Klucz obcy (Foreign Key)

Klucz obcy to kolumna, która wskazuje na `PRIMARY KEY` w innej tabeli. Dzięki niemu łączysz rekordy z dwóch tabel.

```sql
CREATE TABLE customers (
    id    INT AUTO_INCREMENT PRIMARY KEY,
    name  VARCHAR(100) NOT NULL,
    email VARCHAR(255) NOT NULL UNIQUE
);

CREATE TABLE orders (
    id          INT AUTO_INCREMENT PRIMARY KEY,
    customer_id INT NOT NULL,
    amount      DECIMAL(10, 2) NOT NULL,
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (customer_id) REFERENCES customers(id)
        ON DELETE CASCADE
        ON UPDATE CASCADE
);
```

`FOREIGN KEY (customer_id) REFERENCES customers(id)` = kolumna `orders.customer_id` MUSI wskazywać na istniejący `customers.id`. Bez tego mógłbyś wpisać `customer_id = 99999` do zamówienia, a taki user nie istnieje — niespójność.

### Akcje `ON DELETE` / `ON UPDATE`

Co się stanie, gdy rodzic zostanie usunięty / zaktualizowany?

- **`RESTRICT`** (domyślne) — zablokuj operację, jeśli istnieją dzieci.
- **`CASCADE`** — usuń / zaktualizuj dzieci razem z rodzicem.
- **`SET NULL`** — ustaw `NULL` w dzieciach (wymaga, by kolumna dopuszczała NULL).
- **`NO ACTION`** — jak `RESTRICT`, ale sprawdzane na końcu transakcji.

```sql
-- Usuń usera → usuń wszystkie jego zamówienia
FOREIGN KEY (customer_id) REFERENCES customers(id) ON DELETE CASCADE

-- Usuń usera → zablokuj (dopóki nie usuniesz zamówień)
FOREIGN KEY (customer_id) REFERENCES customers(id) ON DELETE RESTRICT

-- Usuń usera → zamówienia zostają, ale customer_id = NULL
FOREIGN KEY (customer_id) REFERENCES customers(id) ON DELETE SET NULL
```

## Typy relacji

### Jeden do wielu (1:N) — najczęstsza

Jeden rekord w tabeli A → wiele rekordów w tabeli B.

```
customers                orders
+----+----------+        +----+-------------+--------+
| id | name     |        | id | customer_id | amount |
+----+----------+        +----+-------------+--------+
|  1 | Anna     |◄───────|  1 |      1      | 50.00  |
|  2 | Jan      |◄───────|  2 |      1      | 30.00  |
+----+----------+        +----+-------------+--------+
                              │       2      | 120.00 | (Jan)
                              └──────────────+--------+
```

Klucz obcy jest po stronie "wielu" (orders). Realizacja: `orders.customer_id → customers.id`.

### Wiele do wielu (N:M) — tabela łącząca

Jeden rekord w A → wiele w B, **i na odwrót**. Realizacja: dodatkowa tabela łącząca.

Przykład: studenci i kursy. Każdy student może być na wielu kursach, każdy kurs ma wielu studentów.

```sql
CREATE TABLE students (
    id   INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL
);

CREATE TABLE courses (
    id    INT AUTO_INCREMENT PRIMARY KEY,
    title VARCHAR(200) NOT NULL
);

-- Tabela łącząca
CREATE TABLE enrollments (
    student_id INT NOT NULL,
    course_id  INT NOT NULL,
    enrolled_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES students(id) ON DELETE CASCADE,
    FOREIGN KEY (course_id) REFERENCES courses(id) ON DELETE CASCADE
);
```

Zapis `INSERT INTO enrollments (student_id, course_id) VALUES (1, 5)` = student 1 zapisany na kurs 5.

### Jeden do jednego (1:1) — rzadka

Np. user i jego profil. Realizacja: klucz obcy z `UNIQUE`.

```sql
CREATE TABLE profiles (
    id      INT PRIMARY KEY,
    bio     TEXT,
    avatar  VARCHAR(255),
    FOREIGN KEY (id) REFERENCES users(id) ON DELETE CASCADE
);
```

## JOIN — łączenie tabel

`JOIN` łączy wiersze z dwóch (lub więcej) tabel na podstawie warunku.

### INNER JOIN — tylko pasujące

```sql
SELECT customers.name, orders.amount, orders.created_at
FROM customers
INNER JOIN orders ON customers.id = orders.customer_id
ORDER BY orders.created_at DESC;
```

Wynik:

| name   | amount | created_at          |
|--------|--------|---------------------|
| Anna   | 50.00  | 2026-05-30 10:00:00 |
| Anna   | 30.00  | 2026-05-29 15:30:00 |
| Jan    | 120.00 | 2026-05-28 09:00:00 |

`INNER JOIN` zwraca **tylko te wiersze, dla których istnieje dopasowanie**. Anna ma 2 zamówienia, Jan 1. Wszyscy userzy bez zamówień **nie pojawiają się**.

### LEFT JOIN — wszyscy z lewej + pasujący

```sql
SELECT customers.name, COUNT(orders.id) AS liczba_zamowien
FROM customers
LEFT JOIN orders ON customers.id = orders.customer_id
GROUP BY customers.id, customers.name
ORDER BY liczba_zamowien DESC;
```

Wynik:

| name   | liczba_zamowien |
|--------|------------------|
| Anna   | 2                |
| Jan    | 1                |
| Ola    | 0                |  ← LEFT JOIN pokazuje Olę mimo braku zamówień

`LEFT JOIN` zwraca **wszystkie wiersze z lewej tabeli** + dopasowane z prawej. Gdy brak dopasowania, prawe kolumny = `NULL`.

### RIGHT JOIN — odwrotność LEFT

`RIGHT JOIN` zwraca wszystkie z prawej + dopasowane z lewej. Rzadko używany — wygodniej zamienić kolejność tabel i użyć LEFT.

### FULL OUTER JOIN — wszystko z obu (MySQL 8+)

```sql
SELECT * FROM customers
FULL OUTER JOIN orders ON customers.id = orders.customer_id;
```

Zwraca wszystkie wiersze z obu tabel. W MySQL domyślnie nieobsługiwane (trzeba emulować `UNION` dwóch LEFT JOIN-ów).

<Callout type="tip" title="Który JOIN?">
- `INNER JOIN` — "pokaż mi tylko tych, co mają powiązanie".
- `LEFT JOIN` — "pokaż wszystkich z lewej, nawet bez powiązania".
- `RIGHT JOIN` — rzadko. Zamień kolejność i użyj LEFT.
- `FULL OUTER JOIN` — "wszystko z obu stron". Używaj rzadko.
</Callout>

### Składnia JOIN — pełna

```sql
SELECT [DISTINCT] kolumny
FROM tabela1
[JOIN rodzaj] tabela2 ON warunek
[JOIN rodzaj] tabela3 ON warunek
WHERE filtr
GROUP BY kolumny
HAVING filtr_grup
ORDER BY kolumny [ASC|DESC]
LIMIT N OFFSET M;
```

## JOIN 3 tabel — przykład

```sql
SELECT
    c.name AS klient,
    o.id AS zamowienie_id,
    p.name AS produkt,
    oi.quantity AS ilosc,
    oi.price AS cena
FROM customers c
INNER JOIN orders o ON c.id = o.customer_id
INNER JOIN order_items oi ON o.id = oi.order_id
INNER JOIN products p ON oi.product_id = p.id
WHERE c.email = 'anna@example.com'
ORDER BY o.created_at DESC;
```

`c`, `o`, `oi`, `p` to **aliasy** — krótsze niż pełne nazwy. Aliasy definiujesz w `FROM customers c` (bez `AS`).

## Samo-złączenie (self join)

Tabela połączona z samą sobą — przydatne np. do hierarchii (pracownik → manager):

```sql
SELECT
    e.name AS pracownik,
    m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;
```

## Podsumowanie

Klucz obcy (`FOREIGN KEY`) łączy tabele i gwarantuje spójność. `ON DELETE CASCADE` usuwa dzieci z rodzicem, `RESTRICT` blokuje, `SET NULL` zeruje. Relacja 1:N (najczęstsza) = klucz obcy po stronie "wielu". N:M = tabela łącząca z dwoma FK. `INNER JOIN` = tylko pasujące. `LEFT JOIN` = wszyscy z lewej + dopasowani. `RIGHT JOIN` = odwrotność. Aliasy (`SELECT ... FROM tabela t`) skracają składnię. JOIN może łączyć 3, 4, 5 tabel naraz — ale im więcej, tym wolniej. Indeksy na kluczach obcych (`FOREIGN KEY (customer_id)` → `CREATE INDEX idx_orders_customer ON orders(customer_id)`) przyspieszają JOIN.
