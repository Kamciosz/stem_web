Wyobraź sobie, że budujesz sklep internetowy. Produkty, zamówienia, użytkownicy — to wszystko musi gdzieś żyć między wizytami. Plik tekstowy odpada przy tysiącu rekordów. Potrzebujesz bazy danych. <Term word="PHP">PHP</Term> bez bazy to jak kuchnia bez lodówki — niby działa, ale nic nie przechowasz na dłużej.

W tej lekcji nauczysz się łączyć PHP z MySQL przez PDO i wykonywać wszystkie cztery operacje <Term word="CRUD">CRUD</Term> (Create, Read, Update, Delete) w jedyny bezpieczny sposób — przez <Term word="prepared statements">prepared statements</Term>.

## PDO vs mysqli — co wybrać

PHP ma dwa interfejsy do baz danych. Zapamiętaj, który jest lepszy:

- **mysqli** — stary, działa tylko z MySQL.
- **PDO** (PHP Data Objects) — nowszy, działa z MySQL, PostgreSQL, SQLite i innymi. **Używaj PDO.**

PDO ma czytelniejszy interfejs do prepared statements i nie przywiązuje Cię do jednej bazy. Jeśli kiedyś zmienisz MySQL na coś innego, zmienisz tylko jeden wiersz w konfiguracji.

## Połączenie z bazą

Zacznij od pliku `db.php`, który będziesz dołączać przez `require_once` do każdego skryptu wymagającego bazy:

```php
<?php
// db.php — współdzielony plik połączenia
$host = "localhost";
$dbname = "sklep";
$user = "root";
$pass = "";

try {
    $pdo = new PDO(
        "mysql:host=$host;dbname=$dbname;charset=utf8mb4",
        $user,
        $pass,
        [
            PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION,  // rzucaj wyjątki
            PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC,  // zwracaj tablice asocjacyjne
            PDO::ATTR_EMULATE_PREPARES => false  // prawdziwe prepared statements
        ]
    );
} catch (PDOException $e) {
    die("Błąd połączenia: " . htmlspecialchars($e->getMessage()));
}
```

Trzy flagi, które musisz rozumieć:

- `ERRMODE_EXCEPTION` — zamiast cichych warningów dostajesz wyjątki. Łatwiej debugować, bo błąd nie przemknie niezauważony.
- `FETCH_ASSOC` — `fetch()` zwraca `["kolumna" => wartość]` zamiast tablicy indeksowanej. Czytelniejszy kod.
- `EMULATE_PREPARES => false` — PDO wysyła prepared statements prawdziwie do MySQL, nie emuluje ich po stronie PHP. To ważne dla bezpieczeństwa.

<Callout type="tip" title="require_once">
Dołączaj plik połączenia przez `require_once` — po pierwszym wczytaniu kolejne wywołania są ignorowane. Nie ma ryzyka podwójnego połączenia ani nadpisania zmiennej `$pdo`.
</Callout>

## Tworzenie tabeli

Zanim zaczniesz pisać PHP, stwórz tabelę w MySQL. Zwróć uwagę na `password_hash` — hasło nigdy nie trafia do bazy jako zwykły tekst:

```sql
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    email VARCHAR(255) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    name VARCHAR(100) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## Dlaczego prepared statements — to jest serce tej lekcji

Zanim przejdziesz do kodu CRUD, zrozum, dlaczego prepared statements są obowiązkowe. Oto co się dzieje bez nich:

```php
// ❌ SQL injection — NIGDY tak nie rób
$email = $_POST["email"];
$stmt = $pdo->query("SELECT * FROM users WHERE email = '$email'");
// User wpisuje: ' OR '1'='1 → zapytanie zwraca WSZYSTKICH userów
// User wpisuje: '; DROP TABLE users; -- → usuwa całą tabelę
```

Atakujący wpisuje w pole formularza fragment <Term word="SQL">SQL</Term>, który zmienia sens całego zapytania. To właśnie <Term word="SQL injection">SQL injection</Term> — jeden z najczęstszych ataków na aplikacje webowe.

Rozwiązanie: oddziel kod SQL od danych. `prepare()` wysyła szablon zapytania do MySQL, `execute()` wstawia dane osobno. MySQL traktuje je jako wartość, nie jako część kodu:

```php
// ✅ Prepared statement — bezpiecznie
$stmt = $pdo->prepare("SELECT * FROM users WHERE email = :email");
$stmt->execute([":email" => $_POST["email"]]);
```

Nawet jeśli ktoś wpisze `' OR '1'='1`, MySQL potraktuje to jako dosłowny ciąg znaków szukany w kolumnie email — nie jako kod SQL.

<Callout type="exam">
Na egzaminie INF.03 każde zapytanie z danymi od użytkownika musi używać prepared statements. Sklejanie stringa SQL z `$_POST` lub `$_GET` to błąd bezpieczeństwa, który kosztuje punkty. Wzorzec: `prepare()` + `execute([":param" => $wartosc])`.
</Callout>

## INSERT — dodawanie danych

Dodajemy nowego użytkownika. Hasło hashujemy **przed** zapisem — do bazy trafia tylko skrót, nigdy oryginał:

```php
function utworzUsera(PDO $pdo, string $email, string $haslo, string $imie): int {
    $hash = password_hash($haslo, PASSWORD_DEFAULT);

    $stmt = $pdo->prepare("
        INSERT INTO users (email, password_hash, name)
        VALUES (:email, :password_hash, :name)
    ");

    $stmt->execute([
        ":email" => $email,
        ":password_hash" => $hash,
        ":name" => $imie
    ]);

    return (int) $pdo->lastInsertId();
}
```

`lastInsertId()` zwraca ID właśnie wstawionego <Term word="rekord">rekordu</Term> — przydatne, gdy od razu chcesz coś z nim zrobić.

## SELECT — pobieranie danych

### Pojedynczy rekord

```php
function znajdzUseraPoEmail(PDO $pdo, string $email): ?array {
    $stmt = $pdo->prepare("
        SELECT id, email, password_hash, name
        FROM users
        WHERE email = :email
        LIMIT 1
    ");
    $stmt->execute([":email" => $email]);
    $user = $stmt->fetch();

    return $user ?: null;
}
```

`fetch()` zwraca jeden wiersz jako tablicę asocjacyjną (lub `false` gdy brak). `LIMIT 1` przyspiesza zapytanie — MySQL przestaje szukać po znalezieniu pierwszego dopasowania.

### Wiele rekordów

```php
function wszyscyUserzy(PDO $pdo): array {
    $stmt = $pdo->query("SELECT id, email, name FROM users ORDER BY created_at DESC");
    return $stmt->fetchAll();
}

foreach (wszyscyUserzy($pdo) as $user) {
    echo htmlspecialchars($user["name"]) . " — " . htmlspecialchars($user["email"]) . "<br>";
}
```

`query()` używaj **tylko dla stałych zapytań** — bez żadnych danych od użytkownika. Gdy tylko pojawia się zmienna od usera, wracasz do `prepare()` + `execute()`.

### Filtrowanie

```php
function szukajUserow(PDO $pdo, string $szukane): array {
    $stmt = $pdo->prepare("
        SELECT id, name, email
        FROM users
        WHERE name LIKE :szukane OR email LIKE :szukane
        ORDER BY name
    ");
    $stmt->execute([":szukane" => "%$szukane%"]);
    return $stmt->fetchAll();
}
```

`%` to wildcard — `%Anna%` znajdzie wszystko zawierające "Anna". Zwróć uwagę: `%` doklejasz do wartości w PHP, nie w SQL. Dzięki temu prepared statement nadal działa poprawnie.

## UPDATE — aktualizacja

```php
function zmienEmail(PDO $pdo, int $id, string $nowyEmail): bool {
    $stmt = $pdo->prepare("UPDATE users SET email = :email WHERE id = :id");
    return $stmt->execute([
        ":email" => $nowyEmail,
        ":id" => $id
    ]);
}
```

`execute()` zwraca `true`/`false`. Jeśli chcesz wiedzieć, czy cokolwiek faktycznie się zmieniło, sprawdź `rowCount()`:

```php
$stmt->execute([":email" => $nowyEmail, ":id" => $id]);
if ($stmt->rowCount() === 0) {
    // Nic nie zaktualizowano — user nie istnieje lub email był taki sam
}
```

## DELETE — usuwanie

```php
function usunUsera(PDO $pdo, int $id): bool {
    $stmt = $pdo->prepare("DELETE FROM users WHERE id = :id");
    $stmt->execute([":id" => $id]);
    return $stmt->rowCount() > 0;
}
```

<Callout type="warning" title="DELETE jest nieodwracalny">
Przed usuwaniem w aplikacji zawsze pytaj użytkownika o potwierdzenie ("Czy na pewno?"). W poważniejszych projektach stosuje się miękkie usuwanie — kolumna `deleted_at` zamiast fizycznego DELETE. Łatwiej cofnąć, łatwiej audytować.
</Callout>

## Transakcje — albo wszystko, albo nic

Wyobraź sobie przelew bankowy: odejmujesz pieniądze z konta A i dodajesz do konta B. Co jeśli aplikacja padnie między tymi dwoma operacjami? Pieniądze znikają. Transakcja rozwiązuje ten problem — albo obie operacje się powiodą, albo żadna:

```php
function przeniesSrodki(PDO $pdo, int $zId, int $doId, float $kwota): void {
    $pdo->beginTransaction();

    try {
        // 1. Odejmij
        $stmt = $pdo->prepare("UPDATE konta SET saldo = saldo - :kwota WHERE id = :id");
        $stmt->execute([":kwota" => $kwota, ":id" => $zId]);

        // 2. Dodaj
        $stmt = $pdo->prepare("UPDATE konta SET saldo = saldo + :kwota WHERE id = :id");
        $stmt->execute([":kwota" => $kwota, ":id" => $doId]);

        $pdo->commit();
    } catch (Exception $e) {
        $pdo->rollBack();
        throw $e;
    }
}
```

`commit()` zatwierdza obie zmiany naraz. `rollBack()` cofa wszystko do stanu sprzed `beginTransaction()`. Bez transakcji awaria między krokiem 1 a 2 zostawia bazę w niespójnym stanie.

## Kompletny przykład — prosty CRUD

Wszystko razem: formularz, INSERT, SELECT, wyświetlanie z `htmlspecialchars()`:

```php
<?php
require_once "db.php";

// CREATE
if ($_SERVER["REQUEST_METHOD"] === "POST" && ($_POST["akcja"] ?? "") === "dodaj") {
    $stmt = $pdo->prepare("INSERT INTO users (email, password_hash, name) VALUES (:e, :p, :n)");
    $stmt->execute([
        ":e" => $_POST["email"],
        ":p" => password_hash($_POST["password"], PASSWORD_DEFAULT),
        ":n" => $_POST["name"]
    ]);
    header("Location: /users.php");
    exit;
}

// READ
$stmt = $pdo->query("SELECT id, email, name, created_at FROM users ORDER BY id");
$users = $stmt->fetchAll();
?>

<h1>Użytkownicy</h1>

<form method="post">
    <input name="email" type="email" placeholder="E-mail" required>
    <input name="password" type="password" placeholder="Hasło" required>
    <input name="name" type="text" placeholder="Imię" required>
    <input type="hidden" name="akcja" value="dodaj">
    <button>Dodaj</button>
</form>

<table>
    <tr><th>ID</th><th>Email</th><th>Imię</th><th>Utworzony</th></tr>
    <?php foreach ($users as $u): ?>
        <tr>
            <td><?= (int)$u["id"] ?></td>
            <td><?= htmlspecialchars($u["email"]) ?></td>
            <td><?= htmlspecialchars($u["name"]) ?></td>
            <td><?= htmlspecialchars($u["created_at"]) ?></td>
        </tr>
    <?php endforeach; ?>
</table>
```

Zwróć uwagę na dwie rzeczy: `password_hash()` przy INSERT i `htmlspecialchars()` przy każdym wyświetleniu. To nie jest opcja — to obowiązek.

## Podsumowanie

PDO + prepared statements to jedyny bezpieczny sposób na <Term word="SQL">SQL</Term> w PHP. Wzorzec: `prepare("SQL z :param")` + `execute([":param" => $wartosc])`. `fetch()` zwraca jeden wiersz, `fetchAll()` — wszystkie. `lastInsertId()` daje ID ostatniego INSERT. `rowCount()` mówi ile wierszy dotknięto przy UPDATE/DELETE. Transakcje (`beginTransaction` / `commit` / `rollBack`) gwarantują atomowość grupy operacji. **Zawsze** `htmlspecialchars()` przy wyświetlaniu danych z bazy — to ochrona przed <Term word="XSS">XSS</Term>. **Zawsze** `password_hash` + `password_verify` do haseł. **Nigdy** nie wstawiaj danych od usera do <Term word="query">query</Term> bez prepared statements — to <Term word="SQL injection">SQL injection</Term>.
