Wchodzisz na Facebooka, logujesz się, przeglądasz posty, klikasz w profil znajomego — i przez cały czas strona wie, że to Ty. Jak? Przecież każde kliknięcie to osobne żądanie <Term word="HTTP">HTTP</Term>, a HTTP z natury nic nie pamięta. Odpowiedź to <Term word="sesja">sesja</Term>. To mechanizm, który daje serwerowi pamięć między żądaniami.

W tej lekcji nauczysz się logować użytkownika, chronić strony przed nieautoryzowanym dostępem i zabezpieczać sesję przed atakami. To materiał egzaminacyjny — i codzienna robota każdego <Term word="backend">backend</Term>-developera.

## Problem: HTTP jest bezstanowy

<Term word="HTTP">HTTP</Term> nie pamięta, kto właśnie odwiedził stronę. Każde żądanie (request) jest niezależne — <Term word="serwer">serwer</Term> nie wie, że to ten sam użytkownik, który 2 sekundy temu pobrał `index.php`.

Gdyby nie było sesji:

- Nie dałoby się zrobić logowania — serwer zapomina po każdym żądaniu.
- Koszyk w sklepie resetowałby się przy każdym kliknięciu.
- Preferencje (motyw, język) musiałyby być przesyłane w każdym URL-u.

## Jak działa sesja — krok po kroku

Sesja to plik (lub wpis w pamięci) na serwerze, przypisany do konkretnego użytkownika. Przeglądarka dostaje **identyfikator sesji** jako <Term word="cookie">cookie</Term> o nazwie `PHPSESSID` i wysyła go z każdym kolejnym żądaniem. Serwer po tym identyfikatorze odnajduje właściwe dane.

```
1. User loguje się → serwer tworzy sesję, zapisuje $_SESSION["user_id"] = 42
2. Serwer wysyła Set-Cookie: PHPSESSID=abc123
3. Przeglądarka zapisuje ciasteczko PHPSESSID=abc123
4. Przy następnym żądaniu przeglądarka wysyła Cookie: PHPSESSID=abc123
5. Serwer widzi PHPSESSID, ładuje $_SESSION["user_id"] = 42
6. User zalogowany ✓
```

Klucz: dane sesji żyją **na serwerze**. Przeglądarka trzyma tylko identyfikator — nie same dane. Dlatego sesja jest bezpieczniejsza niż trzymanie wszystkiego w cookie.

## Włączenie sesji

Na każdej stronie korzystającej z sesji wywołaj `session_start()` — **na samym początku pliku, przed jakimkolwiek wyjściem** (echo, HTML, nawet pusta linia przed `<?php`):

```php
<?php
session_start();    // MUSI być przed wyjściem

// teraz można używać $_SESSION
$_SESSION["counter"] = ($_SESSION["counter"] ?? 0) + 1;
echo "Odwiedziłeś tę stronę {$_SESSION['counter']} razy.";
?>
```

Jeśli w pliku jest `echo` albo `<html>` PRZED `session_start()` — dostaniesz błąd "headers already sent". Nagłówki HTTP (w tym cookie sesji) muszą być wysłane przed treścią.

## Zapis i odczyt z sesji

`$_SESSION` to zwykła tablica PHP. Możesz w niej trzymać cokolwiek — stringi, liczby, tablice:

```php
<?php
session_start();

// Zapis
$_SESSION["user_id"] = 42;
$_SESSION["user_name"] = "Anna";
$_SESSION["koszyk"] = [1, 5, 12];

// Odczyt
if (isset($_SESSION["user_id"])) {
    echo "Zalogowany jako: " . htmlspecialchars($_SESSION["user_name"]);
}

// Usuwanie konkretnego klucza
unset($_SESSION["koszyk"]);

// Sprawdzanie czy klucz istnieje
if (array_key_exists("user_id", $_SESSION)) {
    // ...
}
```

<Callout type="tip" title="Co trzymać w sesji">
W sesji trzymaj `user_id` i metadane (imię, rola). **Nigdy nie trzymaj hasła** — nawet zahashowanego. Hasło żyje tylko w bazie. Dzięki temu nawet wyciek sesji nie oznacza wycieku hasła.
</Callout>

## Logowanie — kompletny przykład

### Formularz (login.php)

```html
<form action="login.php" method="post">
    <label for="email">E-mail</label>
    <input id="email" name="email" type="email" required>

    <label for="pass">Hasło</label>
    <input id="pass" name="password" type="password" required>

    <button type="submit">Zaloguj</button>
</form>
```

### Obsługa logowania

Teraz serce systemu. Zwróć uwagę na każdy krok — każdy ma swój powód:

```php
<?php
require_once "db.php";
session_start();

if ($_SERVER["REQUEST_METHOD"] !== "POST") {
    // Pokaż formularz
    include "login-form.php";
    exit;
}

$email = $_POST["email"] ?? "";
$haslo = $_POST["password"] ?? "";

if (empty($email) || empty($haslo)) {
    $error = "Uzupełnij e-mail i hasło.";
    include "login-form.php";
    exit;
}

$stmt = $pdo->prepare("
    SELECT id, email, password_hash, name
    FROM users
    WHERE email = :email
    LIMIT 1
");
$stmt->execute([":email" => $email]);
$user = $stmt->fetch();

if (!$user || !password_verify($haslo, $user["password_hash"])) {
    $error = "Nieprawidłowy e-mail lub hasło.";
    include "login-form.php";
    exit;
}

// Sukces — ustaw sesję
$_SESSION["user_id"] = $user["id"];
$_SESSION["user_name"] = $user["name"];
$_SESSION["user_email"] = $user["email"];

// Regeneruj ID sesji (zabezpieczenie przed session fixation)
session_regenerate_id(true);

header("Location: /panel.php");
exit;
```

Dlaczego `session_regenerate_id(true)` po zalogowaniu? Bez tego atakujący mógłby podsunąć ofierze znany sobie identyfikator sesji (np. przez link), poczekać aż się zaloguje, a potem użyć tego samego ID do przejęcia konta. Po regeneracji stary ID przestaje działać.

<Callout type="exam">
Na egzaminie INF.03 sprawdzają: `password_verify()` do weryfikacji hasła (nie porównanie stringów!), `session_regenerate_id(true)` po zalogowaniu, `htmlspecialchars()` przy wyświetlaniu danych z sesji. Pominięcie któregokolwiek to błąd bezpieczeństwa.
</Callout>

## Wylogowanie

Wylogowanie to trzy kroki: wyczyść dane sesji, usuń cookie po stronie klienta, zniszcz sesję po stronie serwera. Wszystkie trzy — nie jeden, nie dwa:

```php
<?php
session_start();

// Opcjonalnie: tylko dla zalogowanych
if (!isset($_SESSION["user_id"])) {
    header("Location: /login.php");
    exit;
}

// Wyczyść dane sesji
$_SESSION = [];

// Usuń ciasteczko sesji po stronie klienta
if (ini_get("session.use_cookies")) {
    $params = session_get_cookie_params();
    setcookie(
        session_name(),
        "",
        time() - 42000,
        $params["path"],
        $params["domain"],
        $params["secure"],
        $params["httponly"]
    );
}

// Zniszcz sesję po stronie serwera
session_destroy();

header("Location: /");
exit;
```

## Zabezpieczanie stron — wzorzec guard

Masz panel użytkownika, który powinien być dostępny tylko po zalogowaniu. Zamiast kopiować sprawdzenie do każdego pliku, stwórz jeden plik `auth.php` z funkcją-strażnikiem:

```php
<?php
// auth.php
session_start();

function wymagajLogowania(): array {
    if (!isset($_SESSION["user_id"])) {
        // Zapamiętaj dokąd user chciał iść
        $_SESSION["redirect_after_login"] = $_SERVER["REQUEST_URI"];
        header("Location: /login.php");
        exit;
    }
    return [
        "id" => $_SESSION["user_id"],
        "name" => $_SESSION["user_name"] ?? "",
        "email" => $_SESSION["user_email"] ?? ""
    ];
}

function wymagajAdmina(): void {
    $user = wymagajLogowania();
    if (($user["role"] ?? "user") !== "admin") {
        http_response_code(403);
        die("Brak dostępu");
    }
}
```

Użycie — dwie linijki na początku każdej chronionej strony:

```php
<?php
// panel.php
require_once "auth.php";
$user = wymagajLogowania();

echo "Witaj, " . htmlspecialchars($user["name"]);
?>

<h1>Panel użytkownika</h1>
```

```php
<?php
// admin.php
require_once "auth.php";
wymagajAdmina();

echo "Panel administratora";
?>
```

## Bezpieczeństwo sesji — cztery zasady

### 1. Session fixation — regeneruj ID po logowaniu

```php
session_regenerate_id(true);
```

Po zalogowaniu i po każdej zmianie uprawnień — nowy ID. Stary natychmiast nieważny.

### 2. Session hijacking — zabezpiecz cookie

Atakujący może próbować wykraść `PHPSESSID` przez <Term word="XSS">XSS</Term> lub podsłuch sieci. Trzy flagi cookie to blokują:

```php
session_set_cookie_params([
    "lifetime" => 0,
    "path" => "/",
    "domain" => "",
    "secure" => true,      // tylko HTTPS
    "httponly" => true,    // niedostępne dla JavaScript
    "samesite" => "Lax"   // nie wysyłane cross-site
]);
session_start();
```

`httponly` sprawia, że JavaScript nie może odczytać cookie — nawet jeśli atakujący wstrzyknie skrypt przez XSS, nie wykradnie identyfikatora sesji. `secure` wymusza HTTPS — cookie nie leci przez niezaszyfrowane połączenie.

<Callout type="warning" title="Kolejność ma znaczenie">
`session_set_cookie_params()` musi być wywołane **przed** `session_start()`. Po odwróceniu kolejności ustawienia nie mają żadnego efektu — sesja już wystartowała ze starymi parametrami.
</Callout>

### 3. Session timeout — wyloguj po bezczynności

Użytkownik wychodzi od komputera bez wylogowania. Po 30 minutach bezczynności sesja powinna wygasnąć sama:

```php
session_start();

$timeout = 1800; // 30 minut

if (isset($_SESSION["last_activity"])) {
    if (time() - $_SESSION["last_activity"] > $timeout) {
        // Sesja wygasła — wyloguj
        $_SESSION = [];
        session_destroy();
        header("Location: /login.php?expired=1");
        exit;
    }
}

$_SESSION["last_activity"] = time();
```

### 4. NIE przechowuj haseł w sesji

W sesji trzymaj `user_id` i ewentualnie `name` / `email`. Hasło (zahashowane) żyje **tylko w bazie**. Nawet wyciek pliku sesji nie oznacza wtedy wycieku haseł.

## Kompletny przykład — logowanie z pełnym zabezpieczeniem

```php
<?php
// login.php
require_once "db.php";
session_set_cookie_params(["secure" => true, "httponly" => true, "samesite" => "Lax"]);
session_start();

$error = null;

if ($_SERVER["REQUEST_METHOD"] === "POST") {
    $email = trim($_POST["email"] ?? "");
    $haslo = $_POST["password"] ?? "";

    if ($email === "" || $haslo === "") {
        $error = "Uzupełnij pola.";
    } else {
        $stmt = $pdo->prepare("SELECT id, name, email, password_hash FROM users WHERE email = :e LIMIT 1");
        $stmt->execute([":e" => $email]);
        $user = $stmt->fetch();

        if ($user && password_verify($haslo, $user["password_hash"])) {
            session_regenerate_id(true);
            $_SESSION["user_id"] = $user["id"];
            $_SESSION["user_name"] = $user["name"];

            $redirect = $_SESSION["redirect_after_login"] ?? "/";
            unset($_SESSION["redirect_after_login"]);

            header("Location: $redirect");
            exit;
        } else {
            $error = "Nieprawidłowy e-mail lub hasło.";
        }
    }
}
?>
<!DOCTYPE html>
<html lang="pl">
<head><meta charset="UTF-8"><title>Logowanie</title></head>
<body>
    <h1>Logowanie</h1>
    <?php if ($error): ?>
        <p style="color:red"><?= htmlspecialchars($error) ?></p>
    <?php endif; ?>
    <form method="post">
        <input name="email" type="email" placeholder="E-mail" required>
        <input name="password" type="password" placeholder="Hasło" required>
        <button>Zaloguj</button>
    </form>
</body>
</html>
```

## Podsumowanie

`session_start()` musi być na początku pliku, przed jakimkolwiek wyjściem. `$_SESSION` to zwykła tablica — trzymaj w niej `user_id` i metadane, nigdy hasła. Po udanym logowaniu zawsze `session_regenerate_id(true)` — to ochrona przed session fixation. `password_verify($haslo, $hash)` do weryfikacji, nie porównanie stringów. Wylogowanie = `$_SESSION = []` + `session_destroy()` + usunięcie <Term word="cookie">cookie</Term>. Guard = funkcja `wymagajLogowania()` na początku każdej chronionej strony. Cookie sesji: `httponly`, `secure`, `samesite=Lax` — ustaw przed `session_start()`. Timeout 30 minut chroni przed zostawionym komputerem. Dane z sesji wyświetlaj przez `htmlspecialchars()` — to nadal dane, którym nie ufasz w stu procentach.
