## Co rozwiązuje ten arkusz

System informacji dla maturzystów. Dwie strony: index.php (lista uczniów jako odnośniki + statystyki w 4 blokach) i wynik.php (wyniki wybranego maturzysty).

**Baza danych:** `matura` — tabele: `maturzysta` (id, imie, nazwisko, szkola), `arkusz` (id, rok, przedmiot, sesja), `wyniki` (id, id_maturzysta FK, id_arkusz FK, punkty).

## Jak rozbić wymagania

1. Wypisz dane wejściowe: baza, tabele, obrazy, formularze i pliki wynikowe.
2. Oddziel rzeczy statyczne od dynamicznych. Statyczne robisz w HTML/CSS, dynamiczne w PHP albo JavaScript.
3. Każde zapytanie uruchom osobno przed wklejeniem do kodu. Jeśli SQL nie działa w bazie, nie zacznie działać magicznie w PHP.
4. Zachowaj nazwy z polecenia. Inna nazwa pliku, pola formularza albo tabeli to nie wariant — to błąd.

<Callout type="warning" title="Za co bym obciął">
Za pisanie zapytań 'na oko', bez sprawdzenia wyniku. Za brak klucza obcego, złą nazwę kolumny, dopisanie sortowania którego nie było w poleceniu albo pominięcie ograniczenia LIMIT/WHERE.
</Callout>

## Zapytania SQL — gotowe rozwiązania

Baza: `matura`. Plik SQL: `egzamin.sql`.

### Zapytanie 1: uczniowie ze szkoły T3

```sql
SELECT id, imie, nazwisko FROM maturzysta
WHERE szkola = 'T3'
ORDER BY nazwisko ASC;
```

### Zapytanie 2: unikalne przedmioty

```sql
SELECT DISTINCT przedmiot FROM arkusz;
```

### Zapytanie 3: najstarszy i najmłodszy arkusz

```sql
SELECT MIN(rok) AS 'Najstarszy', MAX(rok) AS 'Najmłodszy' FROM arkusz;
```

### Zapytanie 4: maturzysta z najlepszą średnią

```sql
SELECT id_maturzysta, AVG(punkty) AS Wynik
FROM wyniki
GROUP BY id_maturzysta
ORDER BY Wynik DESC
LIMIT 1;
```

<Callout type="exam">
Zapytanie 4 łączy GROUP BY + ORDER BY + LIMIT — trzy klauzule naraz. Alias `Wynik` w ORDER BY działa w MySQL/MariaDB (w innych bazach nie zawsze). Na egzaminie INF.03 używasz MySQL, więc jest OK.
</Callout>

### Zapytanie 5: wyniki maturzysty id=31 z JOIN

```sql
SELECT a.rok, a.sesja, a.przedmiot, w.punkty
FROM wyniki w
JOIN arkusz a ON w.id_arkusz = a.id
WHERE w.id_maturzysta = 31;
```

## Jak myśli egzaminator przy danych

Egzaminator nie szuka pięknego opisu bazy. Szuka dowodu, że rozumiesz zależność między poleceniem a kodem:

- jeśli polecenie wymaga listy z bazy, musi istnieć zapytanie zwracające dokładnie te pola;
- jeśli wynik ma być ograniczony, w SQL musi być warunek albo `LIMIT`, a nie ręczne ukrywanie danych w HTML;
- jeśli dane pochodzą z formularza, nazwa pola formularza musi zgadzać się z tym, co później czyta PHP albo JavaScript;
- jeśli arkusz wymaga zapisu danych, `INSERT` ma dopisywać rekord do właściwej tabeli, bez ręcznego wymuszania klucza głównego;
- jeśli arkusz nie ma klasycznej bazy, 'danymi' są pliki, nazwy funkcji, identyfikatory i parametry z polecenia.

<Callout type="exam" title="Test zrozumienia">
Po tym etapie powinieneś umieć powiedzieć jednym zdaniem: „Ten arkusz pobiera dane stąd, przetwarza je tak, a wynik pokazuje tutaj”. Jeżeli nie umiesz, to jeszcze nie rozwiązałeś zadania — tylko przepisałeś fragmenty.
</Callout>

## Minimalny dowód poprawności

Przed przejściem do następnego etapu zrób trzy rzeczy:

1. Uruchom każde zapytanie albo sprawdź każdy plik wejściowy osobno.
2. Zanotuj, który fragment strony korzysta z którego wyniku.
3. Wskaż miejsce, gdzie może polecieć punkt: zła nazwa pola, zły typ danych, brak relacji, brak warunku albo zła ścieżka pliku.

To jest nudne, ale działa. Losowe pisanie kodu bez tej mapy kończy się debugowaniem wszystkiego naraz.

## Jak sprawdzić ten etap

- wynik każdego zapytania ma dokładnie te kolumny, których żąda polecenie;
- liczba rekordów zgadza się z warunkiem `LIMIT`, `WHERE` albo agregacją;
- w zapytaniach z więcej niż jedną tabelą jest jawny warunek połączenia;
- zapytanie modyfikujące dane nie wpisuje ręcznie wartości, które baza ma nadać automatycznie;
- zrzuty ekranów/import bazy robisz dopiero po sprawdzeniu wyniku, nie przed.
